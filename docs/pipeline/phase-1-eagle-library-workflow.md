# Phase 1: Eagle Library Workflow

## 문서 목적

이 문서는 Reflix 파이프라인의 Phase 1 운영 절차를 정리한다. 목표는 두 가지다.

- 지금 당장 사람이 문서만 보고 작업 순서를 따라갈 수 있어야 한다.
- 나중에 자동화할 때 구현 기준으로 그대로 옮길 수 있어야 한다.

즉 이 문서는 운영 매뉴얼이면서 자동화 요구사항 문서다.

## Phase 1 범위

Phase 1은 원본 mp4를 Eagle 라이브러리에 정리하고, 이름/태그/태그 그룹/폴더/썸네일과 AI 메타데이터를 정리해 `배포 후보` 상태로 만드는 단계다.

이 문서는 Reflix 앱 배포 자체를 다루지 않는다. 배포는 Phase 2의 범위다.

## 시작 조건

Phase 1은 아래 조건 중 하나라도 있으면 시작한다.

- 새 mp4 파일이 수집되었다.
- 기존 mp4 또는 Eagle 아이템 중 이름/태그/폴더/썸네일/AI 메타데이터 정리가 필요한 항목이 발견되었다.
- 이전에 `보류` 상태로 남겨둔 항목을 다시 검토해야 한다.

## 종료 조건

아래 조건을 모두 만족하면 해당 아이템은 Phase 1을 통과한 것으로 본다.

- 영상이 Eagle에 들어가 있다.
- 이름이 정리되어 있다.
- 태그가 붙어 있다.
- 새 태그 여부와 태그 그룹 상태가 정리되어 있다.
- 폴더가 분류되어 있다.
- 썸네일이 정리되어 있다.
- Eagle metadata에 `aiTags` 필드가 존재하며, 값이 객체 또는 `null`로 정리되어 있다.
- 따라서 Reflix 배포 후보로 올릴 수 있다.

## 공통 상태

각 아이템은 아래 메인 상태 중 하나를 가진다.

- `수집됨 (collected)`
- `이름 정리 완료 (named)`
- `Eagle 저장 완료 (imported_to_eagle)`
- `썸네일 처리 완료 (thumbnail_ready)`
- `태그 정리 완료 (tagged)`
- `폴더 정리 완료 (foldered)`
- `배포 후보 (publish_candidate)`
- `보류 (on_hold)`

이 메인 상태는 “어디까지 왔는가”를 나타낸다.

## 공통 플래그

메인 상태와 별개로, 검토가 필요한 문제를 플래그로 표시한다. 하나의 아이템에 여러 플래그가 동시에 붙을 수 있다.

- `중복 의심 (possible_duplicate)`
- `이름 애매 (name_needs_review)`
- `새 태그 후보 (new_tag_candidate)`
- `태그 그룹 미분류 (tag_group_unassigned)`
- `폴더 미분류 (folder_unassigned)`
- `썸네일 처리 실패 (thumbnail_failed)`
- `AI 메타데이터 보강 필요 (ai_metadata_missing)`
- `수동 검토 필요 (manual_review_required)`

## 공통 운영 원칙

### 1. 중복 의심 항목은 임의로 삭제하지 않는다

중복처럼 보여도 바로 삭제하지 않는다. 최소한 아래 중 하나로 분류한다.

- 중복 아님
- 중복 의심
- 대표본 결정 필요

### 2. 애매한 항목은 억지로 확정하지 않는다

이름, 태그, 폴더가 애매한 항목은 `보류` 또는 관련 플래그로 남긴다. 잘못된 확정은 나중에 더 큰 정리 비용을 만든다.

### 3. Eagle 내부 작업과 Reflix 배포 작업은 섞지 않는다

Phase 1은 Eagle 라이브러리 품질 관리 단계다. Reflix 앱에 실제로 무엇을 올릴지는 Phase 2에서 `release batch`로 결정한다.

### 4. 현재 상태와 목표 상태를 같이 기록한다

각 단계에는 “지금은 어떻게 하고 있는지”와 “나중에 자동화할 목표가 무엇인지”를 함께 적는다.

### 5. 숫자 토큰은 태그로 붙이지 않는다

`(2)`, `2`, `03` 같은 숫자 또는 숫자 중심 토큰은 태그로 사용하지 않는다. 이런 값은 보통 시퀀스 표기, 임시 정리 흔적, 파일 버전 표시에 가깝고 검색 의미를 거의 만들지 못한다.

### 6. AI 메타데이터는 Eagle metadata에 먼저 쓴다

`src/data/index.json`과 `public/data/clips/*.json`은 export 산출물이다. AI 태깅은 이 파생 파일에 직접 덧쓰지 않고, Eagle 원본 `metadata.json`에 `aiTags`를 기록한 뒤 export에서 그대로 흘려보내는 순서를 유지한다.

## 공통 규칙: 중복 영상 검사

중복 검사는 하나의 단계에만 속하지 않는다. Phase 1 전체를 관통하는 공통 검수 규칙이다.

### 목적

같은 영상이 여러 번 들어가거나, 사실상 같은 영상이 다른 이름으로 중복 관리되는 일을 막는다.

### 검사 시점

- mp4 수집 시 1차 검사
- 이름 정리 후 2차 검사
- Eagle import 후 3차 확인

### 중복 의심 기준

- 파일명만 다르고 내용이 사실상 같은 경우
- 같은 행동/장면인데 길이만 조금 다른 경우
- 이미 Eagle에 들어간 아이템과 사실상 같은 경우

### 처리 원칙

- 중복 확정 전까지는 삭제하지 않는다.
- `possible_duplicate` 플래그를 붙인다.
- 대표본을 결정할 근거가 부족하면 `manual_review_required`로 넘긴다.

## 1. mp4 수집

### 단계 목적

새 mp4를 Phase 1 작업 대상으로 등록한다.

### 현재 상태

사람이 새로 들어온 mp4를 폴더에 저장하고, 작업 대상으로 삼을 파일을 직접 골라낸다.

### 목표 상태

신규 mp4를 자동으로 감지하거나 일괄 스캔해, 작업 대상 목록을 안정적으로 만들 수 있다.

### 입력

- 새로 저장한 mp4 파일

### 작업

- 파일이 실제로 열리는지 확인한다.
- 손상 여부를 확인한다.
- 이번 처리 대상인지 구분한다.
- 기존 파일 또는 Eagle 아이템과 겹치는지 1차 확인한다.

### 판정 기준

- 재생 가능한 mp4다.
- 파일 손상이 없다.
- 이번 작업 대상 목록에 포함할 수 있다.

### 자주 생기는 문제

- 확장자는 mp4지만 실제로 재생이 안 되는 파일
- 이름만 다르고 내용은 같은 파일
- 파일명만 봐서는 내용을 전혀 식별할 수 없는 파일

### 결과물

- 신규 mp4 후보 목록
- 중복 의심 후보 목록

### 다음 단계로 넘기는 조건

- 메인 상태가 `수집됨 (collected)` 이어야 한다.
- 중복 여부가 최소한 1차 분류돼 있어야 한다.

### 관련 코드/문서/경로

- [Reflix 전체 파이프라인 개요](./reflix-pipeline-overview.md)

## 2. 이름 정리

### 단계 목적

영상 내용을 검색 가능한 형태로 정리된 이름으로 만든다.

### 현재 상태

사람이 오탈자, 동의어, 표현 길이를 직접 판단해 이름을 정리한다.

### 목표 상태

자동으로 이름 후보를 제안하고, 사람은 승인/보류만 결정한다.

### 입력

- `수집됨 (collected)` 상태의 신규 mp4

### 작업

- 오탈자를 정리한다.
- 동의어를 운영 기준에 맞게 통일한다.
- 불필요한 표현을 제거한다.
- 이름만 봐도 행동/감정/상황이 어느 정도 드러나는지 확인한다.

### 판정 기준

- 이름만 읽어도 영상의 핵심 내용이 어느 정도 유추된다.
- 팀 기준 용어와 충돌하지 않는다.
- 지나치게 길거나 모호하지 않다.

### 자주 생기는 문제

- 어떤 단어를 표준으로 써야 할지 아직 정해지지 않은 경우
- 이름을 줄이면 의미가 빠지는 경우
- 같은 뜻의 표현이 여러 개 있어서 통일 기준이 흔들리는 경우

### 결과물

- 최종 이름
- 이름 변경 전/후 기록

### 다음 단계로 넘기는 조건

- 메인 상태가 `이름 정리 완료 (named)` 이거나
- `이름 애매 (name_needs_review)` 플래그와 함께 `보류 (on_hold)`로 분류되어야 한다.

### 관련 코드/문서/경로

- [기존 Eagle Phase 2 계획](/Users/macbook/reflix-nextjs/docs/superpowers/plans/2026-03-23-eagle-phase2-name-tag-folder-implementation.md)

### 실제 사례

- `연출 아케인 힘듦 일어나기 비몽사몽 비틀비틀 아픔`
- `연출 소울 페이셜 디테일 감동`

이런 식으로 작품명 + 핵심 상태 + 행동/상황이 이름에 드러나는 방향을 기본으로 삼는다.

## 3. Eagle 저장

### 단계 목적

정리된 mp4를 Eagle 라이브러리에 1개 아이템으로 저장한다.

### 현재 상태

현재는 사람이 Eagle 앱에서 직접 import를 수행하고, 이름과 미리보기가 정상인지 최종 확인한다.

### 목표 상태

가져오기 자체는 자동화하되, 중복과 미리보기 이상 여부는 검토 대상으로 남긴다.

### 입력

- `이름 정리 완료 (named)` 상태의 mp4

### 작업

- Eagle에 import 한다.
- 이름이 의도한 대로 반영되었는지 확인한다.
- 아이템이 열리고 미리보기가 가능한지 확인한다.
- 기존 Eagle 아이템과 중복 import된 것이 없는지 다시 본다.

### 판정 기준

- 원본 mp4와 Eagle 아이템이 1:1로 대응된다.
- 이름이 정리된 상태로 저장되었다.
- Eagle에서 정상적으로 열리고 확인 가능하다.

### 자주 생기는 문제

- 같은 파일이 중복으로 import 됨
- 이름이 Eagle 안에서 의도와 다르게 들어감
- import는 됐지만 미리보기/썸네일이 이상함

### 결과물

- Eagle 신규 아이템 집합

### 다음 단계로 넘기는 조건

- 메인 상태가 `Eagle 저장 완료 (imported_to_eagle)` 이어야 한다.
- 필요하면 `possible_duplicate` 또는 `manual_review_required` 플래그를 붙인다.

### 관련 코드/문서/경로

- [Eagle 라이브러리 경로 해석](/Users/macbook/reflix-nextjs/scripts/lib/eagle-library-path.mjs)

## 4. Eagle 썸네일 처리

### 단계 목적

Eagle 내부에서 아이템을 더 빠르게 식별할 수 있도록 움직이는 썸네일 상태로 정리한다.

### 현재 상태

프로젝트 터미널에서 별도 썸네일 스크립트를 실행해 Eagle 내부 썸네일을 교체하고, 결과 확인은 Eagle 앱에서 한다.

### 목표 상태

대상 조건에 맞는 아이템에 대해 썸네일 처리를 반복 가능하고 안정적으로 수행한다.

### 입력

- `Eagle 저장 완료 (imported_to_eagle)` 상태의 아이템

### 작업

- 프로젝트 루트 터미널에서 Eagle 썸네일 처리 워크플로우를 실행한다.
- 처리 전 백업을 만든다.
- Eagle 앱에서 결과가 의도대로 보이는지 확인한다.

### 판정 기준

- Eagle 내부에서 썸네일이 의도대로 움직이거나 표시된다.
- 실패 항목이 별도로 구분된다.
- 원본 썸네일을 복구할 수 있는 백업이 존재한다.

### 자주 생기는 문제

- 썸네일 파일 구조가 예상과 다름
- 처리 후 Eagle에서 썸네일이 깨짐
- 실패 항목이 누락되어 다음 단계에서 발견됨

### 결과물

- 썸네일 처리 완료 목록
- 썸네일 실패/보류 목록
- 썸네일 백업

### 다음 단계로 넘기는 조건

- 메인 상태가 `썸네일 처리 완료 (thumbnail_ready)` 이거나
- `썸네일 처리 실패 (thumbnail_failed)` 플래그와 함께 `보류 (on_hold)`로 분류되어야 한다.

### 관련 코드/문서/경로

- [Eagle 썸네일 처리 스크립트](/Users/macbook/reflix-nextjs/scripts/eagle-thumbnail-pilot.mjs)
- [미디어 전략 문서](/Users/macbook/reflix-nextjs/docs/media-strategy.md)
- `.tmp/eagle-thumbnail-backups/...`
- 실행 위치: 프로젝트 루트 터미널
- 확인 위치: Eagle 앱

## 5. 태그 부착

### 단계 목적

영상 검색과 필터링에 필요한 의미 태그를 붙인다.

### 현재 상태

사람이 이름과 영상을 보고 핵심 태그를 붙인다.

### 목표 상태

이름과 규칙 기반으로 태그 후보를 만들고, 사람이 예외만 확인한다.

### 입력

- `썸네일 처리 완료 (thumbnail_ready)` 또는 `Eagle 저장 완료 (imported_to_eagle)` 상태의 아이템

### 작업

- 행동, 감정, 상황, 작품명 같은 핵심 태그를 붙인다.
- 중복되거나 의미가 약한 태그를 줄인다.
- 이름과 태그가 어긋나지 않는지 확인한다.
- `(2)`, `2` 같은 숫자 토큰은 태그 후보에서 제외한다.

### 판정 기준

- 검색에 필요한 핵심 태그가 들어 있다.
- 이름과 태그가 서로 크게 어긋나지 않는다.
- 불필요한 태그가 과도하게 붙어 있지 않다.
- 숫자만으로 이루어진 태그나 시퀀스 표기 태그가 붙어 있지 않다.

### 자주 생기는 문제

- 영상 의미가 애매해서 태그 확정이 어려운 경우
- 기존 태그로 충분한지, 새 태그가 필요한지 헷갈리는 경우
- 비슷한 태그가 여러 개 존재하는 경우

### 결과물

- 아이템별 태그 상태
- 신규 태그 후보 목록

### 다음 단계로 넘기는 조건

- 메인 상태가 `태그 정리 완료 (tagged)` 이거나
- `새 태그 후보 (new_tag_candidate)` 플래그를 부착해야 한다.

## 6. 신설 태그 검사 및 태그 그룹 분류

### 단계 목적

새로 생긴 태그가 있는지 확인하고, 있다면 기존 태그 체계 안에서 어느 그룹에 속하는지 정리한다.

### 현재 상태

신규 태그 여부와 그룹 배정은 사람이 검토해서 확정한다.

### 목표 상태

신규 태그 후보를 자동으로 분리하고, 사람이 승인과 그룹 지정만 한다.

### 입력

- `태그 정리 완료 (tagged)` 상태의 아이템
- `새 태그 후보 (new_tag_candidate)` 플래그 목록

### 작업

- 새 태그가 실제 신규인지 확인한다.
- 기존 태그로 대체 가능한지 검토한다.
- 새 태그라면 어느 태그 그룹에 속하는지 정한다.

### 판정 기준

- 신규 여부가 확정되었다.
- 새 태그라면 태그 그룹이 정해졌다.
- 태그 체계와 실제 사용 상태가 크게 어긋나지 않는다.

### 자주 생기는 문제

- 일회성 태그인지 반복 사용될 태그인지 모호한 경우
- 기존 태그와 거의 겹치지만 완전히 같지는 않은 경우
- 어느 그룹에 넣어야 할지 기준이 없는 경우

### 결과물

- 신규 태그 검토 결과
- 태그 그룹 반영 결과

### 다음 단계로 넘기는 조건

- `태그 그룹 미분류 (tag_group_unassigned)` 플래그가 해소되었거나
- 보류 사유가 분명하게 남아 있어야 한다.

## 7. 폴더 미분류 항목 분류

### 단계 목적

Eagle 폴더가 없는 아이템을 적절한 위치에 분류한다.

### 현재 상태

사람이 이름, 태그, 맥락을 보고 폴더를 결정한다.

### 목표 상태

규칙 기반으로 폴더 후보를 만들고, 애매한 항목만 수동 검토한다.

### 입력

- `태그 정리 완료 (tagged)` 상태의 아이템
- 폴더가 비어 있거나, 분류가 불안정한 아이템

### 작업

- 현재 폴더 체계 기준으로 적절한 위치를 찾는다.
- 여러 폴더 후보가 있으면 우선순위를 판단한다.
- 확신이 없으면 보류로 남긴다.

### 판정 기준

- 폴더 없는 항목이 적절한 위치에 들어간다.
- 애매한 항목은 무리하게 확정하지 않는다.
- 폴더 체계와 실제 분류 결과가 크게 어긋나지 않는다.

### 자주 생기는 문제

- 여러 폴더에 동시에 걸치는 경우
- 어느 폴더에도 자연스럽지 않은 경우
- 폴더 체계 자체를 먼저 손봐야 하는 경우

### 결과물

- 폴더 분류 완료 목록
- 미분류/보류 목록

### 다음 단계로 넘기는 조건

- 메인 상태가 `폴더 정리 완료 (foldered)` 이어야 한다.
- 폴더를 확정하지 못한 경우 `폴더 미분류 (folder_unassigned)` 와 함께 `보류 (on_hold)`로 분류한다.

## 8. AI 메타데이터 보강

### 단계 목적

수동 태그를 보강하는 `aiTags`를 Eagle 원본 metadata에 기록해, 이후 export가 검색/인스펙터용 AI 메타데이터를 안정적으로 소비할 수 있게 만든다.

### 현재 상태

정식 백필 파이프라인은 아직 확정 전이고, 파일럿 프롬프트/모델 검증과 수동 확인이 함께 진행되는 단계다.

### 목표 상태

`aiTags`가 없는 항목만 배치로 골라 Vision 모델에 보내고, 결과를 검증 가능한 구조화 JSON으로 Eagle `metadata.json`에 안정적으로 저장한다.

### 입력

- `폴더 정리 완료 (foldered)` 상태의 아이템
- `AI 메타데이터 보강 필요 (ai_metadata_missing)` 플래그 목록
- 기존 이름/수동 태그
- 원본 영상 또는 export용으로 정규화된 입력 미디어

### 작업

- `aiTags` 필드가 없는 항목만 선별한다.
- 기존 이름과 수동 태그를 컨텍스트로 함께 보내 구조화 태그와 자연어 설명을 생성한다.
- 응답 JSON을 정규화하고 필수 키가 모두 있는지 확인한다.
- Eagle `metadata.json`을 백업한 뒤 `aiTags`를 기록한다.
- API 실패나 파싱 실패 시 `aiTags: null`로 저장해 “시도했지만 결과 없음” 상태를 명확히 남긴다.
- 중단 후 재개를 위해 진행 상태와 실패 목록을 별도로 기록한다.

### 판정 기준

- Eagle metadata에 `aiTags` 필드가 존재한다.
- `aiTags` 값이 구조화 객체이거나 `null`이다.
- `generatedAt`, `model` 같은 메타 정보가 함께 남아 있다.
- 원본 metadata를 복구할 수 있는 백업이 존재한다.

### 자주 생기는 문제

- Vision 모델 응답이 JSON 스키마를 어기는 경우
- 짧거나 애매한 클립에서 설명 품질이 흔들리는 경우
- API 실패 때문에 일부 항목만 누락되는 경우
- 파생 JSON에는 값이 보이지만 Eagle 원본 metadata에는 반영되지 않은 경우

### 결과물

- Eagle metadata의 `aiTags` 반영 결과
- AI 태깅 실패/재시도 목록
- metadata 백업
- 진행 상태 체크포인트

### 다음 단계로 넘기는 조건

- `aiTags` 필드가 누락된 항목이 없어야 한다.
- 실패 항목은 `aiTags: null` 또는 명확한 보류 사유로 상태가 설명되어야 한다.
- 이후 해당 아이템을 `배포 후보 (publish_candidate)` 로 올릴 수 있어야 한다.

## Phase 1 최종 체크리스트

아래 체크리스트를 통과하면 해당 아이템은 `배포 후보 (publish_candidate)` 로 올릴 수 있다.

- Eagle에 1:1로 저장되어 있다.
- 이름이 정리되어 있다.
- 썸네일이 정리되어 있다.
- 태그가 붙어 있다.
- 새 태그와 태그 그룹 문제가 통제되고 있다.
- 폴더가 정리되어 있다.
- Eagle metadata의 `aiTags` 필드가 객체 또는 `null`로 정리되어 있다.
- 중복 또는 애매한 항목이 통제되고 있다.

## Phase 2로 넘기는 기준

Phase 1을 통과한 아이템은 바로 배포되는 것이 아니다. 다음 조건을 만족한 뒤, Phase 2의 `release batch` 후보로 넘긴다.

- 메인 상태가 `배포 후보 (publish_candidate)` 이다.
- 치명적인 검토 플래그가 남아 있지 않다.
- `aiTags` 필드가 없는 상태로 남아 있지 않다.
- 남아 있는 플래그가 있다면 운영자가 이유를 이해하고 관리할 수 있다.

즉 Phase 1의 목표는 “모든 원본을 곧바로 배포한다”가 아니라, “Eagle 라이브러리 기준으로 배포 가능한 품질 상태를 만든다”이다.
