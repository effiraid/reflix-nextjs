# Reflix — Eagle 2단계 이름/태그/폴더 정리 설계

## 개요

Reflix의 2단계 작업은 Eagle 라이브러리의 `분류되지 않은 아이템`을 정리하는 운영 파이프라인이다. 대상은 바탕화면 라이브러리 `레퍼런스 - 게임,연출.library` 안에서 `folders`가 비어 있는 항목만이며, 현재 기준 수량은 `874개`다.

이 단계의 목표는 세 가지다.

1. 아이템 이름의 표기 이상 후보를 검토 가능한 목록으로 추출한다.
2. 승인된 이름을 기준으로 태그를 이름 단어와 `1:1`로 맞춘다.
3. 별도 규칙표를 이용해 각 아이템을 적절한 폴더에 다중 배치한다.

이 작업은 실제 Eagle 항목 메타 파일인 `images/<id>.info/metadata.json`을 읽고 갱신하는 방식으로 수행한다.

## 목표와 비목표

### 목표

- `folders`가 비어 있는 대상만 처리한다.
- 이름 수정은 자동 반영 전에 반드시 검토 목록으로 제시한다.
- 태그는 승인된 최종 이름을 기준으로 기계적으로 동기화한다.
- 폴더는 일반 단어 자동 추론이 아니라 명시적 규칙표 기반으로 배치한다.
- 실행 전후 차이를 추적할 수 있는 리포트와 백업을 남긴다.

### 비목표

- 이미 폴더가 있는 항목을 재분류하지 않는다.
- 의미 확장 태그를 유지하지 않는다. 이름에 없는 태그는 제거 대상이다.
- 폴더를 이름 토큰 전체로 자동 생성하거나 자동 추론하지 않는다.
- 웹 앱의 런타임, 공개 데이터 포맷, 프론트엔드 UI는 이번 단계에서 변경하지 않는다.

## 현재 확인된 사실

- 대상 수: `874개`
- 이름 단어와 태그가 이미 정확히 일치하는 항목은 일부에 불과하다.
- 현재 태그에는 이름에 없는 확장 태그가 많이 섞여 있다.
  - 예: `놀라움`, `불안`, `우울`, `피`, `손`, `발`, `검`, `시전`
- 폴더 이름 토큰만으로 자동 매칭하면 과매칭이 발생한다.
  - 예: `게임`, `연출`, `전투`, `레이아웃` 같은 일반 단어가 거의 모든 후보를 여러 폴더에 걸치게 만든다.
- 따라서 태그와 폴더는 서로 다른 규칙으로 처리해야 한다.

## 설계 원칙

- 이름 검토는 사람 승인 전까지 읽기 전용이다.
- 태그는 이름 기준 단순 동기화다.
- 폴더는 별도 규칙표 기반이다.
- 실패 시 즉시 원복 가능해야 한다.
- 결과는 반복 실행 가능해야 한다.

## 전체 구조

파이프라인은 여섯 개 역할로 분리한다.

### 1. 수집기

Eagle 라이브러리에서 대상 항목만 읽는다.

- 입력: Eagle `.library` 경로
- 필터:
  - `metadata.json` 존재
  - `isDeleted === false`
  - `ext === "mp4"`
  - `folders.length === 0`
- 수집 필드:
  - `id`
  - `name`
  - `tags`
  - `folders`
  - `metadataPath`
  - `_mediaPath`
  - `_thumbnailPath`
  - `_infoDir`

### 2. 이름 검토기

이름 수정이 필요해 보이는 항목만 후보로 추린다. 이 단계에서는 Eagle 메타데이터를 수정하지 않는다.

### 3. 반영기

사용자가 승인한 이름 수정 목록만 실제 `metadata.json`의 `name` 필드에 반영한다.

### 4. 태그 동기화기

최종 이름을 기준으로 태그를 재구성한다.

### 5. 폴더 배치기

별도 규칙표를 읽어 폴더 ID를 계산하고 `folders` 배열을 갱신한다.

### 6. 리포터

검토용, 규칙용, 실제 반영 결과용 파일을 남긴다.

## 이름 검토 설계

이름 수정은 전수 수정보다 `후보 추출 + 검토 승인` 방식으로 간다. 핵심은 의미 변경이 아니라 표기 이상 탐지다.

### 후보 추출 규칙

#### 기계적 이상

- 괄호 숫자 접미
  - 예: `(2)`, `(3)`
- 불균형 괄호
- 연속 공백
- 동일 단어 반복
- 앞뒤 불필요한 구두점

#### 표기 일관성 이탈

같은 시리즈나 같은 접두 패턴을 가진 이름끼리 비교했을 때, 대부분과 다른 철자를 쓰는 항목을 후보로 올린다.

예:
- 대다수는 `스타세일러`인데 일부만 다른 철자를 쓰는 경우
- 같은 묶음 안에서 작품명/클래스명 표기가 하나만 튀는 경우

#### 희귀 토큰 이탈

전체 대상에서 거의 한 번만 나오고, 자주 등장하는 다른 토큰과 편집 거리가 매우 가까운 단어를 후보로 올린다. 단, 실제로 자주 쓰는 토큰은 예외 목록으로 둔다.

예외 목록 예시:
- `33원정대`
- `2d`
- `pov`
- `pv`

### 검토 산출물

이름 검토 단계는 두 파일을 남긴다.

- `name-review.md`
  - 사람이 검토하기 쉬운 표 형식
- `name-review.json`
  - 자동 반영 입력으로 재사용 가능한 구조화 형식

권장 필드:

- `id`
- `currentName`
- `proposedName`
- `reason`
- `confidence`
- `approved`

`approved`는 기본값이 `false`거나 비어 있어야 하며, 사용자 검토 후 승인된 항목만 반영 단계에 입력된다.

## 태그 동기화 설계

태그는 승인된 최종 이름을 `trim` 후 공백 기준으로 분해한 결과를 원본으로 사용한다. 의미 확장, 동의어 추가, 그룹 태그 보강은 하지 않는다.

### 태그 생성 규칙

- 이름 단어는 기본적으로 모두 태그 후보가 된다.
- 다음 토큰은 태그에서 제외한다.
  - 순수 숫자
    - 예: `2`, `3`, `4`
  - 괄호 숫자
    - 예: `(2)`, `(3)`
- 다음 토큰은 그대로 유지한다.
  - `33원정대`
  - `2d`
  - `pov`
  - `pv`
  - `오브`
  - `손`
  - `발`

### 태그 반영 규칙

- 현재 태그 중 이름에 없는 태그는 제거한다.
- 이름에 있는 태그 후보 중 현재 태그에 없는 것은 추가한다.
- 최종 상태는 `이름 단어 - 숫자 예외 = 태그`가 되도록 만든다.

중복 단어가 있을 때 Eagle이 동일 태그 중복 저장을 허용하지 않으면, 실제 저장 결과는 유일 집합이 될 수 있다. 이 경우 해당 항목은 결과 리포트에 `lossyTagSync`로 표시한다.

## 폴더 배치 설계

폴더 배치는 이름 전체 토큰을 직접 폴더에 매핑하지 않는다. 별도 규칙표에 정의된 토큰만 폴더 계산에 사용한다.

### 왜 규칙표가 필요한가

폴더 이름 토큰 자동 매칭은 과매칭을 일으킨다.

- `게임`
- `연출`
- `전투`
- `레이아웃`

이런 일반 단어는 분류력이 낮고 여러 폴더를 동시에 가리키므로, 자동 배치 기준으로 쓰면 안 된다.

### 규칙표 원칙

- 규칙표는 토큰 기반이다.
- 하나의 토큰이 여러 폴더를 가리킬 수 있다.
- 규칙에 없는 토큰은 폴더 계산에 사용하지 않는다.
- 폴더는 다중 배치를 허용한다.
- 폴더 ID는 [categories.json](/Users/macbook/reflix-nextjs/src/data/categories.json) 의 실제 폴더 ID를 기준으로 저장한다.

### 규칙 예시

- 작품/세계관
  - `원신`
  - `명조`
  - `갓오브워`
  - `스타세일러`
- 클래스/장비
  - `마법사`
  - `검`
  - `둔기`
  - `활`
- 동작/상태
  - `밀치기`
  - `착지`
  - `승리`
  - `시전`

### 규칙표 산출물

초기 구현은 별도 JSON 파일로 관리한다. 예:

- `scripts/config/eagle-phase2-folder-rules.json`

필드 예시:

- `token`
- `folderIds`
- `note`

## 실행 흐름

실행은 두 단계 명령으로 분리한다.

### 1. 검토 단계

읽기 전용으로 수행한다.

- 대상 `874개` 수집
- 이름 수정 후보 생성
- 폴더 규칙 미매칭 토큰 보고서 생성
- 현재 이름/태그/폴더 스냅샷 저장

이 단계에서는 Eagle `metadata.json`을 수정하지 않는다.

### 2. 반영 단계

승인된 이름 수정 목록을 입력으로 받아 실제 변경을 수행한다.

순서:

1. 백업 생성
2. 승인된 이름 반영
3. 이름 기준 태그 동기화
4. 규칙표 기준 폴더 계산 및 반영
5. 후속 검증
6. 결과 리포트 저장

## 산출물과 백업

모든 산출물은 프로젝트 아래 `.tmp/`에 남긴다.

예:

- `.tmp/eagle-phase2/<timestamp>/name-review.md`
- `.tmp/eagle-phase2/<timestamp>/name-review.json`
- `.tmp/eagle-phase2/<timestamp>/folder-rule-report.json`
- `.tmp/eagle-phase2/<timestamp>/apply-report.json`
- `.tmp/eagle-phase2-backups/<timestamp>/<id>.metadata.json`

백업은 각 아이템의 원본 `metadata.json`을 그대로 복사한 형태여야 한다.

## 실패 처리와 원복

- 개별 항목 반영 실패 시 해당 항목은 즉시 백업본으로 원복한다.
- 나머지 항목 처리는 계속할 수 있지만, 실패 항목은 `apply-report.json`에 명시한다.
- 다수 실패 시 전체 실행을 중단하고 백업 디렉터리를 기준으로 수동 롤백 가능해야 한다.

## 검증 기준

반영 단계 종료 후 다음을 검증한다.

- 이름 검토 승인 목록이 정확히 반영되었는가
- 이름 단어와 태그 차집합이 `0`인가
  - 단, 숫자/괄호 숫자 토큰은 비교 대상에서 제외
- 규칙표에 의해 계산된 폴더가 `folders`에 반영되었는가
- 폴더가 여전히 비어 있는 항목 수가 얼마인가
- `lossyTagSync`가 발생한 항목이 있는가

## 제안 파일 구조

- `scripts/eagle-phase2.mjs`
  - 검토/반영 모드 CLI 진입점
- `scripts/lib/eagle-phase2.mjs`
  - 수집, 이름 후보 추출, 태그 동기화, 폴더 계산, 리포트 생성
- `scripts/config/eagle-phase2-folder-rules.json`
  - 토큰 기반 폴더 매핑 규칙

기존 [eagle-reader.mjs](/Users/macbook/reflix-nextjs/scripts/lib/eagle-reader.mjs) 는 읽기 전용 수집에 재사용하고, 쓰기 로직은 phase2 전용 모듈에서 분리한다.

## 구현 전 고정된 결정사항

- 대상은 `folders`가 비어 있는 항목만
- 이름 수정은 검토 후 반영
- 태그는 이름 공백 단어 기준
- 순수 숫자와 괄호 숫자는 태그 제외
- 조각처럼 보이는 토큰도 태그 유지
- 폴더는 다중 배치 허용
- 폴더는 규칙표 기반
